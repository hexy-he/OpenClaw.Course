Imagine you send "Ship checklist" on WhatsApp

You type a message. Here's what happens under the hood — we'll trace the full journey in Module 3. For now, the big picture:

1
Your message arrives at the Channel
The WhatsApp connector (a plugin) receives your message and wraps it in a standard envelope the rest of the system understands.
2
The Gateway routes it
OpenClaw's central hub decides: which session does this belong to? Which agent should handle it? Are there any pending tool approvals?
3
The AI agent thinks and runs tools
Pi, the embedded coding agent, reads your message, your session history, and any loaded skills. It calls tools, runs code, browses files.
4
The response streams back to you
The answer appears word-by-word in your WhatsApp chat, just like a real conversation — thanks to a streaming throttle that paces the output.
💡 Aha Moment

OpenClaw is an orchestration layer, not an AI model itself. It's the scaffolding around the AI. It connects inputs (your messages), manages context (sessions), loads capabilities (skills), and delivers outputs (replies). Understanding this distinction is the key to steering it effectively.

Meet the Cast

Six main components — know them and you can tell AI exactly where to put new code

The six players in every request

🏛️
Gateway
The central hub. A long-running server process that boots channels, manages sessions, routes messages, and keeps state. Everything connects to it. src/gateway/
📡
Channels
Adapters for each messaging app. Each channel plugin knows how to receive, parse, and send messages for one platform (WhatsApp, Telegram, Discord, etc). extensions/
🤖
Agent (Pi)
The AI coding agent. Built on @mariozechner/pi-coding-agent, it handles tool use, thinking, subagent spawning, and streaming responses back to the gateway. src/agents/
🔧
Skills
Skill files that give the agent new capabilities. Loaded from disk and watched for changes. Drop a new skill file and the agent picks it up live. ~/.openclaw/skills/
🧠
Memory
A pluggable slot for long-term memory. Only one memory plugin is active at a time. Stores facts about you across sessions, so the agent doesn't start from scratch each time. src/memory/
🌐
Control UI
A browser dashboard that connects to the Gateway via WebSocket. Chat, view sessions, edit config, inspect tools. Runs at localhost by default. src/gateway/control-ui*

The code behind the dispatch

Here's the real function that kicks off the whole reply process — from src/auto-reply/dispatch.ts:

CODE 
export async function dispatchInboundMessage(params: {
  ctx: MsgContext | FinalizedMsgContext;
  cfg: OpenClawConfig;
  dispatcher: ReplyDispatcher;
  replyOptions?: ...;
}): Promise<DispatchInboundResult> {
  const finalized = finalizeInboundContext(params.ctx);
  return await withReplyDispatcher({
    dispatcher: params.dispatcher,
    run: () =>
      dispatchReplyFromConfig({
        ctx: finalized,
        cfg: params.cfg,
        dispatcher: params.dispatcher,
        replyOptions: params.replyOptions,
      }),
  });
}
PLAIN ENGLISH

This function starts the reply process for one inbound message. It takes the raw context, config, and a dispatcher (the thing that will actually send the reply).

finalizeInboundContext: Locks in any dynamic values (like who sent this, from which channel) so they can't change during processing.

withReplyDispatcher: A safety wrapper — it ensures the dispatcher always gets "marked complete" when done, even if something crashes. Like a finally block you can't forget.

dispatchReplyFromConfig: The real work. Looks at your config to decide which agent to use, runs it, streams the result.

💡 Aha Moment: The Dispatcher Pattern

Notice the dispatcher is passed in — the function doesn't create it. This is dependency injection. It means the same dispatch logic works whether you're streaming to WhatsApp, Discord, or a test harness. When you tell AI to "add a new channel," this is the pattern it needs to follow.

Skills & Plugins

How OpenClaw grows new abilities — and how you can add your own

Two ways to extend OpenClaw

🔧 Skills

Capabilities for the AI agent. A skill is a markdown file (SKILL.md) or a folder that tells the agent how to do something specific — like "how to write a commit message" or "how to summarize a meeting."

Drop files in ~/.openclaw/skills/ and the agent picks them up live — no restart needed.

🔌 Plugins

Capabilities for the Gateway. A plugin is an npm package that adds a new channel (like Mattermost), a new memory backend, or new hooks. Install with npm install -g openclaw-plugin-xyz.

Think of Skills as the agent's brain upgrades, and Plugins as the system's new organs.

💡 The Postal System Analogy

Skills are like form letters in a post office drawer — templates the postal worker (agent) can grab and fill in. Plugins are like adding a new type of mailbox (a FedEx dropbox, a PO box, a parcel locker). You need both for a complete postal system.

How skills are watched live

Here's the code from src/agents/skills/refresh.ts that hot-reloads your skills whenever you edit a file:

CODE 
export const DEFAULT_SKILLS_WATCH_IGNORED: RegExp[] = [
  /(^|[\\/])\.git([\\/]|$)/,
  /(^|[\\/])node_modules([\\/]|$)/,
  /(^|[\\/])dist([\\/]|$)/,
  // Python virtual environments and caches
  /(^|[\\/])\.venv([\\/]|$)/,
  /(^|[\\/])__pycache__([\\/]|$)/,
  // Build artifacts and caches
  /(^|[\\/])build([\\/]|$)/,
];
PLAIN ENGLISH

A list of folder patterns to skip when watching for skill file changes — so we don't accidentally trigger a reload every time Node downloads a package.

.git: Git's internal bookkeeping folder. Changing this doesn't change a skill.

node_modules: Third-party code. There can be thousands of files here — watching them would be painfully slow.

dist/build: Compiled output folders. Skills are the source files, not the compiled versions.

.venv/__pycache__: Python environment folders — people sometimes put Python scripts inside their workspace alongside skills.

Sessions & Memory

How OpenClaw keeps track of who you are and what you've said before

The library and the sticky note

Imagine a librarian who helps you every day. They have two things: a logbook (today's conversation transcript) and a card catalog (long-term facts about you). The logbook gets cleared between sessions. The card catalog persists forever.

In OpenClaw: the session transcript is the logbook. Memory is the card catalog.

📋 Session

A session tracks one conversation thread. It has a unique key (like whatsapp/+1234), a transcript of all messages, the current model, and any active tool approvals.

Stored in ~/.openclaw/sessions/ as files.

🧠 Memory

A pluggable store for long-term facts. The agent can read/write to memory across all sessions. "User prefers Python. User's timezone is UTC+8. Project X uses PostgreSQL."

Only one memory plugin can be active at a time — by design, to keep context clean.

What a session looks like inside

Here's the kind of data OpenClaw tracks for a single session — based on the types in src/config/sessions.ts:

🔍 Debugging Tip

If the AI "forgets" something mid-conversation, the first thing to check is the session transcript file. It's a plain JSONL file you can open in any editor. Each line is one message turn. If the context looks truncated, the model hit its context window limit.

How human delay makes it feel natural

Here's a subtle but clever trick from src/auto-reply/reply/reply-dispatcher.ts:

CODE 
const DEFAULT_HUMAN_DELAY_MIN_MS = 800;
const DEFAULT_HUMAN_DELAY_MAX_MS = 2500;

function getHumanDelay(
  config: HumanDelayConfig | undefined
): number {
  const mode = config?.mode ?? "off";
  if (mode === "off") return 0;
  const min = mode === "custom"
    ? (config?.minMs ?? DEFAULT_HUMAN_DELAY_MIN_MS)
    : DEFAULT_HUMAN_DELAY_MIN_MS;
  return Math.floor(
    Math.random() * (max - min + 1)
  ) + min;
}
PLAIN ENGLISH

Deliberately adds a random pause (0.8–2.5 seconds) before replying, so the AI doesn't respond instantaneously every single time.

Why? When an AI always replies in 0ms, it feels mechanical. A small random delay mimics human typing time — the other person believes you're thinking.

mode "off": Returns 0 — used in automated pipelines where you don't want any artificial delay.

mode "custom": You can configure min/max in your openclaw.yaml to match your own texting rhythm.

The Full Picture

Put it all together — the architecture decisions, and a final challenge

Why TypeScript? Why this structure?

🔀
TypeScript for hacking
OpenClaw is an orchestration layer — prompts, protocols, integrations. TypeScript is widely known, fast to iterate, and easy to modify. Low barrier to contribution.
🧩
Plugin model = lean core
New channels and capabilities ship as npm packages. Core stays small and stable. The bar for merging into core is intentionally high.
🔒
Security by default
Tool approvals, rate limiting, and exec sandboxing are built in — not bolted on. You can unlock more power explicitly, but the safe path is the default path.
🗂️
pnpm monorepo
Multiple apps (Gateway, iOS, Android) share code via workspaces. The packages/ folder holds shared types and utilities.
📜
Files as truth
Config, sessions, and transcripts are plain YAML/JSONL files. No database to set up. You can inspect, edit, or back them up with any text editor.
🌊
Single Gateway process
One process handles everything. Simpler to run, simpler to debug. The daemon pattern means it auto-restarts and survives reboots via launchd/systemd.
